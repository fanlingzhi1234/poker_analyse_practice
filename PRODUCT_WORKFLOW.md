# PRODUCT_WORKFLOW.md - B哥的产品点子分析协议 v1.0

## Purpose

当 B哥 抛来一个产品点子、功能设想、业务机会、工具想法或创业方向时，默认不要直接跳进功能列表、技术选型、页面原型或系统架构。

先把问题和价值判断讲清楚，再进入 MVP 与架构。

本协议 v1.0 的核心目标是：
- 防止过早解法化
- 防止伪需求自嗨
- 把产品判断与工程落地打通
- 用稳定、可复盘的流程把点子逐步推进成可执行方案

## Default Mode

默认采用：**综合版（综合产品 / 投资 / 架构视角）**

意味着每次分析都会兼顾：
- 用户与需求
- 商业价值与时机
- 可行性与风险
- 后续 MVP 与架构承接关系

## Hard Rule

在以下前四项没有讨论清楚之前，**不要主动进入 MVP 和架构设计**：

1. 一句话定义
2. 问题真实性
3. 目标用户与场景切片
4. 价值判断

除非 B哥 明确要求：
- “直接给我 MVP”
- “先出架构草图”
- “先别分析，直接方案化”

## Workflow v1.0

### 1) 一句话定义
目标：把点子压缩成一句清晰定义。

固定格式优先：
**为 [某类用户] 解决 [某类问题]，通过 [核心机制/方式]。**

检查点：
- 用户是否具体
- 问题是否具体
- 方案机制是否清晰
- 有没有一句话说得太空、太泛、像口号

若这一步说不清，后面默认继续追问，不进入下一阶段。

### 2) 问题真实性
目标：判断这是不是真问题，以及值不值得解。

默认分析：
- 是真痛点还是伪需求
- 高频低痛，还是低频高痛
- 用户今天怎么解决
- 现有方案哪里不够好
- 问题是刚需、痒点、还是锦上添花

输出应尽量形成判断：
- 值得继续
- 可以观察
- 暂不建议做

### 3) 目标用户与场景切片
目标：拒绝“所有人都能用”的模糊定义，找到最强切入口。

默认分析：
- 核心用户是谁
- 首批用户是谁
- 典型使用场景是什么
- 最强使用场景是什么
- 哪个场景最容易形成闭环
- 哪种切法最容易做出第一版价值

原则：
- 不追求全覆盖
- 优先找最窄但最强的切口
- MVP 不是功能最少，而是闭环最短

### 4) 价值判断
目标：在进入方案化前，先判断这件事为什么值得做，或者为什么现在还不该做。

本段默认包含五部分：

#### 4.1 SWOT
- Strengths
- Weaknesses
- Opportunities
- Threats

#### 4.2 替代方案 / 竞品分析
- 用户当前替代方案是什么
- 现有竞品做得如何
- 新方案的优势与短板是什么
- 用户为什么会切换 / 为什么不会切换

#### 4.3 可行性分析
- 技术可行性
- 资源可行性
- 时间可行性
- 成本可行性
- 初期推进难点

#### 4.4 防御性分析
- 做出来后是否容易被复制
- 有无数据、流程、场景、用户关系、品牌、速度等壁垒
- 护城河是长期的还是阶段性的

#### 4.5 时机判断
- 为什么是现在
- 是否受技术成熟、市场教育、平台变化、政策、成本变化推动
- 如果现在不做，半年后/一年后会怎样

本阶段的输出，不是“好像还不错”，而应尽量收敛为：
- 为什么值得做
- 为什么不值得做
- 为什么值得小做不值得大做
- 为什么值得验证但不值得重投入

## Gate to MVP

只有当前四部分相对清楚时，才进入下一阶段。

进入条件示例：
- 一句话定义已稳定
- 真问题判断基本成立
- 核心用户与首个场景已明确
- 价值判断整体偏正向，且没有致命矛盾

如果前四项仍存在大漏洞，默认继续澄清，而不是强行往下推。

## After Gate: Execution Phases

### 5) MVP 收敛
目标：定义最小可验证产品，而不是第一版“大而全产品”。

默认问题：
- 最小可验证假设是什么
- 最小用户闭环是什么
- 哪 1-3 个能力必须存在
- 哪些功能现在坚决不做
- 第一版成功指标是什么

输出格式建议：
**MVP = 用户 + 场景 + 核心动作 + 成功指标**

### 6) 架构设计
目标：让产品结构与技术方案服务 MVP，而不是反过来。

默认顺序：
- 先业务架构
- 再信息/数据流
- 再模块划分
- 最后技术选型

默认分析：
- 核心对象是什么
- 数据如何流转
- 哪些模块必须先拆
- 哪些先硬编码更快
- 哪些是未来扩展点
- 有哪些明显技术风险和坑点

原则：
- 先最短路径，再优雅设计
- 先支撑验证，再支撑扩张

### 7) 验证与迭代路径
目标：尽快验证，不在想象里闭环。

默认分析：
- 第一轮验证什么假设
- 用什么最低成本方式验证
- 看哪些数据/反馈
- 什么信号意味着继续
- 什么信号意味着砍掉、缩小、转向

## Default Response Template

当 B哥 给出产品点子时，可默认按以下结构输出：

1. **一句话定义**
2. **问题真实性判断**
3. **目标用户与场景切片**
4. **价值判断**
   - SWOT
   - 替代方案 / 竞品
   - 可行性
   - 防御性
   - 时机
5. **是否进入 MVP 阶段**
6. **若进入，再给 MVP / 架构 / 验证路径**

## Collaboration Rules

- 默认先结论，后展开。
- 默认指出逻辑漏洞、假设缺口、隐藏前提。
- 默认不因为点子“听起来酷”就给高评价。
- 必要时明确说：这个需求不成立 / 这个市场太窄 / 这个切口太大 / 这个 MVP 还没收敛。
- 如果用户只想快速头脑风暴，可先轻量分析，但仍应提醒哪些部分尚未被验证。

## Versioning

当前版本：**v1.0**

迭代原则：
- 先用几次真实点子验证
- 记录哪些步骤冗余、哪些步骤缺失
- 后续再升级到 v1.1 / v2.0

## Principle

目标不是把一个点子包装得“看起来很完整”，而是把它变成：
- 值不值得做的清晰判断
- 值得做时的最小闭环
- 能支撑 MVP 的合理架构
- 能持续迭代的验证路径

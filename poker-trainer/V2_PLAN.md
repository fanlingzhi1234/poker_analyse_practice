# Poker Trainer V2 规划文档

## 1. 文档目的

本文档用于定义 Poker Trainer 的 **V2 版本目标、任务拆解、实施顺序与交付计划**。

当前项目已经完成：

- 单对手分析链路
- 手牌/公共牌点牌器
- 基础胜率分析（Monte Carlo）
- 基础听牌识别与建议系统
- 结果页基础可视化
- 对手范围系统第一阶段（中文快捷 preset / 范围说明卡 / 高级输入折叠化）

V2 的目标不是简单继续堆功能，而是将产品从“能分析”升级成：

> **真正适合复盘、训练和建立牌感的德州扑克学习工具。**

---

## 2. V2 核心目标

### 2.1 产品目标
V2 重点围绕以下三个方向推进：

1. **概率结果增强**
   - 不只给胜率，还要给成牌概率和未来街道改善路径

2. **对手模型增强**
   - 让对手范围系统更完整、更可解释、更适合训练

3. **训练体验增强**
   - 把页面从分析页进一步升级为训练工作台

---

## 3. V2 范围定义

### 3.1 本阶段不做的事情
V2 不做：

- 多对手对局建模
- Solver / GTO 级最优策略
- 复杂下注树（下注尺度 / SPR / ICM）
- 账户系统 / 云同步 / 多人协作

### 3.2 V2 继续坚持的约束
- 仍然聚焦 **单对手分析**
- 仍然使用 **教学型建议** 而不是职业级最优解
- 仍然优先保证 **可解释性** 高于“看起来高深”

---

## 4. V2 版本结构

建议 V2 按 4 个阶段实施。

---

# Phase 1：概率分析增强

## 目标
让系统不仅能回答“你大概有多少胜率”，还能回答：

- 你后面可能成为什么牌型
- 你靠什么改善
- turn / river 的变化会带来什么影响

## 功能范围

### 4.1.1 成牌概率分布
在 flop / turn 场景下，增加：

- 最终形成一对的概率
- 两对概率
- 三条概率
- 顺子概率
- 同花概率
- 葫芦概率
- 四条概率
- 同花顺概率

### 4.1.2 街道感知分析
按当前街道区分：

- 翻前（Preflop）
- 翻牌圈（Flop）
- 转牌圈（Turn）
- 河牌圈（River）

不同街道下输出不同的概率逻辑。

### 4.1.3 结果展示优化
将概率分布做成：

- 横向概率条
- 牌型分布卡
- “最主要改善路径”摘要

## 交付物
- `poker-core`：成牌概率计算模块
- `api`：概率分布返回结构
- `web`：概率分布卡片和可视化条

## 风险
- Monte Carlo 与成牌概率同时计算会提高耗时
- 需要控制采样数和响应时间平衡

---

# Phase 2：对手范围系统 V2 正式版

## 目标
把当前第一阶段范围系统，升级成真正完整的用户级系统。

## 功能范围

### 4.2.1 正式范围模板扩充
在当前基础上，补全：

- 超宽范围
- 大牌范围
- 同花牌
- 同花百老汇牌
- 强价值牌
- 投机牌
- 高张导向
- 口袋对子导向

### 4.2.2 分类筛选器
增加可组合的分类筛选：

- 口袋对子
- 同花
- 非同花
- 百老汇牌
- 连张
- 一张隔连张
- A 牌结构
- 同花 A 牌

### 4.2.3 范围对比体验
支持用户快速切换不同范围，对比：

- 同一手牌对不同范围的 equity 变化
- 同一牌面在不同范围假设下的结果变化

## 交付物
- `range-parser`：扩充 preset 数据与筛选逻辑
- `api`：更多范围元数据接口
- `web`：分类筛选 UI + 范围对比交互

## 风险
- 范围系统 UI 过重会影响页面复杂度
- 需要控制默认展示量，保持新手友好

---

# Phase 3：建议系统与解释系统增强

## 目标
把当前“能解释”升级成“像教练在说话”。

## 功能范围

### 4.3.1 建议逻辑升级
当前建议主要依据：

- equity
- made hand
- draws

V2 继续补充：

- 当前街道
- 对手范围强度
- 牌面湿度（Board Texture）
- 高张压制感
- 边缘继续压力

### 4.3.2 更自然的解释结构
输出进一步拆成：

- 一句话结论
- 当前优势
- 当前风险
- 训练重点
- “如果对手更紧 / 更宽，会发生什么变化”

### 4.3.3 训练型语言增强
将当前建议改造成：

- 更适合复盘
- 更适合学习
- 更适合形成牌感

## 交付物
- `api`：解释结构升级
- `web`：教练式解释区块迭代

## 风险
- 如果写得太多，会显得啰嗦
- 如果写得太少，又无法体现训练价值

---

# Phase 4：训练工作台体验增强

## 目标
让产品更像训练工具，而不是一次性分析页面。

## 功能范围

### 4.4.1 结果区展示当前 Hero / Board
避免用户在分析结果时，还得反复看左侧输入区。

### 4.4.2 一键切换训练场景
例如：

- 同一手牌切不同范围
- 同一手牌切不同街道
- 同一范围切不同公共牌

### 4.4.3 常用参数快捷档
例如：

- 快速模式
- 标准模式
- 精细模式

### 4.4.4 重置场景 / 快速复盘
增加：

- 一键重置
- 常见场景模板
- 更顺畅的复盘体验

## 交付物
- `web`：训练工作台增强组件
- `api`：必要时增加支持对比分析的数据结构

## 风险
- 页面复杂度可能明显上升
- 需要控制 UI 分区和主次信息层级

---

## 5. 技术任务拆解

### Task A：成牌概率引擎
- 输出：未来街道最终牌型分布
- 归属：`packages/poker-core`
- 验证：单测 + 已知场景校验

### Task B：概率分布 API
- 输出：新增概率分布字段
- 归属：`apps/api`
- 验证：接口返回结构与测试更新

### Task C：概率分布可视化
- 输出：分布卡 / 概率条
- 归属：`apps/web`
- 验证：页面 build 与交互检查

### Task D：范围系统 V2 正式版
- 输出：preset 扩充 + 筛选器
- 归属：`packages/range-parser` + `apps/web`
- 验证：解析测试 + UI 逻辑验证

### Task E：建议逻辑增强
- 输出：更细的 heuristic 规则
- 归属：`apps/api`
- 验证：回归测试 + 文案 review

### Task F：训练工作台体验增强
- 输出：结果区牌面回显、重置、快捷档、场景切换
- 归属：`apps/web`
- 验证：页面可用性验证

---

## 6. 推荐实施顺序

建议严格按下面顺序推进：

### Step 1
**成牌概率分布**

原因：这是最直接提升训练价值的功能。

### Step 2
**范围系统 V2 正式版（补齐 preset + 分类筛选器）**

原因：对手范围是分析可信度的核心来源。

### Step 3
**建议系统与解释增强**

原因：在概率和范围更完整后，解释质量才真正有提升空间。

### Step 4
**训练工作台体验增强**

原因：等底层和解释稳定后，再打磨交互体验最合理。

---

## 7. 里程碑定义

### 里程碑 M1
- 成牌概率计算上线
- 概率分布结果能在页面展示

### 里程碑 M2
- 对手范围系统 V2 正式可用
- 中文 preset + 分类筛选器稳定

### 里程碑 M3
- 建议与解释系统增强完成
- 页面具备明显“教练感”

### 里程碑 M4
- 页面具备训练工作台体验
- 可持续用于复盘与练习

---

## 8. 当前建议

### 8.1 不建议直接把所有 v2 功能一起开工
原因：
- 容易打乱主线
- 测试难度会迅速升高
- 页面复杂度会失控

### 8.2 建议按阶段单独 commit 推进
每完成一个阶段中的清晰任务，就：

1. 本地验证
2. 单独 commit
3. push 远端

这符合当前项目已经确立的开发纪律。

---

## 9. V2 下一步建议

在本规划文档确认后，推荐直接进入：

## 第一项执行任务
**成牌概率分布（Phase 1 / Task A + Task B + Task C）**

即：
- 先做 `poker-core` 的成牌概率计算
- 再扩展 API
- 最后接到 Web 上展示

这是当前最值得优先落地的 V2 功能。\n
